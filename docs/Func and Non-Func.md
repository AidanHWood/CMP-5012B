

| ID | Type | Requirement | Priority (MoSCoW) | Justification |
| :---- | :---- | :---- | :---- | :---- |
| FR1 | Functional | The system will allow users to log meals and snacks with nutritional content. | Must Have | The core purpose of the application; supports building healthy eating habits for all users. |
| FR2 | Functional | Users will be allowed to log workouts including type, duration, intensity and calories burned. | Should Have | Important for users tracking fitness but not essential for basic health tracking. |
| FR3 | Functional | The system will display the user's progress over time using charts and summaries. | Must Have | Visual feedback is critical for long term motivation and engagement. |
| FR4 | Functional | The system will allow users to log in, create an account and store personal data securely. | Must Have | Required for personalised tracking over time. |
| FR5 | Functional | The system will send notifications to users (if enabled) to encourage working towards goals. | Should Have | Improves engagement and accountability but not essential for core functionality. |
| FR6 | Functional | The system could support data import from external devices such as smart watches and other health apps. | Could Have | A nice-to-have feature that increases usability but does increase technical complexity |
| FR7 | Functional | The system should give monthly/weekly summaries to users about their progress. | Should Have | Summaries help the user to reflect upon their progress increasing user retention and motivation. |
| NFR1 | Non-Functional | On a standard mobile connection, the application should load within a respectable time (e.g. 3s) | Must Have | Poor performance reduces user retention and usability. |
| NFR2 | Non-Functional | The application should minimise the number of interactions required to log a meal, workout, goal, etc. | Must Have | Ease of use is critical for busy and low-motivation users. |
| NFR3 | Non-Functional | The system will adhere to GDPR and allow users to delete the information we store on them. | Must Have | Legal requirements are essential for trust with sensitive health data. |
| NFR4 | Non-Functional | The system should be modular to allow for new features/changes to be made quickly. | Should Have | Supports long-term maintainability and future expansion. |
| NFR5 | Non-Functional | The application should be built to fit the dimensions of most/all phone models to allow a range of users. | Should Have | Improves inclusivity and user reach but not required. |
| NFR6 | Non-Functional | The system could support offline logging and then synchronisation once connected to wifi. | Could Have | Improves usability in areas of poor connection but would increase technical complexity. |
| NFR7 | Non-Functional | The system should be designed to support scalability to support a larger user base in the future. | Should Have | Supports future growth without hindering performance for new users. |
